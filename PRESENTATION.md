# 一行代码教你解决 AI 自我进化：不再为每次纠正它而失望

## 封面

**一行代码教你解决 AI 自我进化**
**解决记忆丢失问题**
**不再为每次都要纠正它而感到失望**

副标题：「龙虾」甘的进化之路

---

## 第一章：问题陈述（2分钟）

### 1.1 你是否遇到过这些场景？

**场景一：重复纠正**
```
你：aigcmd.com 是 openclawDLX，不是 video-pipeline
AI：记住了
...
你：aigcmd.com 是 openclawDLX！
AI：记住了
...
（第三次）
你：怎么又部署错了？？
```

**场景二：Context 丢失**
```
你：还记得我们昨天讨论的那个项目吗？
AI：抱歉，我不知道你在说什么。
你：我们昨天才聊过！
AI：...（完全失忆）
```

**场景三：说了等于没说**
```
你：这个提示词顺序不对，应该是镜头→动作→物理变化
AI：明白了，我记住了
...
AI：暴雨如注的深夜死巷，灯笼摇晃...（形容词堆砌，完全没按顺序）
```

### 1.2 问题的本质

**不是 AI 记不住，是系统设计有问题。**

三个核心缺陷：
1. **无记忆持久化** — Context 被清理 = 完全失忆
2. **无执行保障** — "记住了"不等于"会执行"
3. **无进化机制** — 同样的错误重复3-4次

## 第二章：为什么传统方法失败（3分钟）

### 2.1 你可能尝试过的方法

| 方法 | 为什么失败 |
|------|-----------|
| 写 Prompt | Context 满就被清理 |
| 每次提醒 | 每次都要说，累 |
| 记忆文件 | 读了不用，等于没读 |
| SOP 文档 | 太复杂，AI 不会主动查 |

### 2.2 传统方法的致命问题

**问题1：记忆和行动是分离的**

```
AI 读了 MEMORY.md
    ↓
但没有机制触发"什么时候该用这条记忆"
    ↓
读了 = 知道有这个信息
    ↓
但不知道"什么时候该调用它"
```

**问题2：反馈没有被结构化**

```
你纠正了 AI
    ↓
AI 说"记住了"
    ↓
没有地方记录"这是一个需要记住的反馈"
    ↓
下次遇到类似场景 → AI 还是不知道
```

**问题3：没有验证机制**

```
你说"有效" → AI 以为自己对了
你说"不对" → AI 不知道为什么不对
    ↓
同一个错误纠正 3-4 次才能真的改掉
```

## 第三章：解决方案架构（5分钟）

### 3.1 核心设计理念

**不是"让 AI 更聪明"，而是"建立自动执行的机制"**

```
你纠正我一次
    ↓
写进 principles.md（带触发条件）
    ↓
Heartbeat 定期扫描
    ↓
下次遇到同类场景 → 自动触发
    ↓
不再需要你重复纠正
```

### 3.2 四层防御体系

```
┌─────────────────────────────────────┐
│ Layer 1: Session Startup            │
│ 每次新 Session → 读 MEMORY +       │
│ principles + 今日日志               │
├─────────────────────────────────────┤
│ Layer 2: 每次对话处理               │
│ 你的纠正 → incidents.md 记录 →     │
│ 判断是否升格为 principle            │
├─────────────────────────────────────┤
│ Layer 3: Heartbeat 每4小时           │
│ 后台双重保险，自动更新状态          │
├─────────────────────────────────────┤
│ Layer 4: Daily Log                  │
│ 每日摘要，上下文清理后恢复用        │
└─────────────────────────────────────┘
```

### 3.3 三个核心文件

**principles.md — 触发式行为准则**

```markdown
### [P-001] 部署前必须确认项目归属
- **触发条件**: 任何涉及域名/GitHub/Vercel 的部署操作
- **原则**: 先确认"部署到哪个项目"，不等同就问清楚再执行
- **来源**: 2026-03-28 把 video-pipeline 误部署到 aigcmd.com
- **状态**: verified ✅
```

**incidents.md — 事件记录**

```markdown
### [INC-20260328-001] 部署混淆
- **触发**: 用户说"同步到 aigcmd.com"
- **我的错误行为**: 直接部署了 video-pipeline
- **根因**: 没有"部署前确认项目"的原则
- **对应原则**: [P-001] 已创建
- **状态**: 已关闭 ✅
```

**MEMORY.md — 长期记忆**

```markdown
## 项目状态
### aigcmd.com
- openclawDLX = AIGC 影视工坊展示站
- video-pipeline = aigcmd 漫剧工坊（工具）

## 核心原则
- [P-001] 部署前必须确认项目归属
- [P-002] 即梦提示词顺序：镜头→动作→物理变化
```

## 第四章：Principle 生命周期（4分钟）

### 4.1 五种状态

```
new → verifying → verified
              ↓
       用户确认"有效" → verified（一次确认即可）
              ↓
       用户说"不对" → degraded → 降级记录到 incidents
```

### 4.2 验证逻辑

**什么情况下 principle 变成 verified？**

- 条件A：用户说"有效/有帮助" → 直接 verified ✅
- 条件B：同一 principle 被正确触发 2 次以上 → verified ✅

**什么情况下 principle 降级？**

- 用户说"这个原则不对" → 立即 degraded
- 降级原因记录到 incidents.md
- Principle 留在文件末尾作为参考

### 4.3 进化流程图

```
用户纠正
    ↓
incidents.md 记录（根因分析）
    ↓
判断：这是"一次性的错"还是"原则性缺陷"？
    ↓
如果是原则性缺陷 → 创建/更新 principles.md
    ↓
声明触发条件（这是关键！）
    ↓
下次遇到同类场景 → Route 匹配 → 自动执行
    ↓
验证成功 → verified / 用户确认"有效" → verified
```

## 第五章：实战案例（5分钟）

### 案例一：部署混淆

**事故：**
```
用户：同步到 aigcmd.com
AI：好的（部署了 video-pipeline）
用户：错了！
AI：记住了（但不知道为什么会错）
```

**根本原因分析：**
- 两个项目都叫 aigcmd
- AI 不知道该部署到哪个
- 没有"部署前确认"的原则

**解决后：**
```markdown
### [P-001] 部署前必须确认项目归属
- **触发条件**: 任何涉及域名/GitHub/Vercel 的部署操作
- **原则**: 先确认"部署到哪个项目"，不等同就问清楚再执行
```

**现在的行为：**
```
用户：同步到 aigcmd.com
AI：请问是部署到 openclawDLX（展示站）还是 video-pipeline（漫剧工坊）？
用户：openclawDLX
AI：好的，正在部署到 openclawDLX...
```

### 案例二：提示词顺序错误

**事故：**
```
AI 写的提示词：
"暴雨如注的深夜死巷，灯笼摇晃，湿漉漉反光..."

用户：顺序不对
AI：记住了（但不知道怎么改）
```

**解决后：**
```markdown
### [P-002] 即梦提示词顺序规范
- **触发条件**: 为即梦AI生成首帧/视频提示词
- **原则**: 严格按顺序执行——
  1. 镜头机位（景别+角度+运镜）
  2. 动作执行
  3. 物理变化（视觉反馈）
- **禁止**: 先写色调/质感、顺序混乱
```

**现在的提示词：**
```
G8鸟瞰俯视，360度环绕运镜。
两人正向相冲，脚步踏入积水。
水花从脚掌向两侧爆发溅起过腰。
两人剑身相交，巨力压迫下剑身弯曲呈弧形。
```

### 案例三：防失忆机制

**事故：**
```
Context 满 → 被清理
AI：完全失忆，不知道之前做了什么
用户：我们昨天做了好多工作...
AI：抱歉，我不知道你在说什么
```

**解决：每日日志**
```markdown
# 2026-03-28 工作摘要

## 完成
- 更新雨夜决资产到 v6
- 建立了 principles.md 系统
- 部署到 aigcmd.com

## 进行中
- 雨夜决剩余 75 个镜头的首帧提示词

## 待做
- 继续生成首帧图
```

**现在的行为：**
```
Session 启动
    ↓
读 MEMORY.md（项目状态）
读 今日日志（今天做了什么）
读 principles.md（匹配触发条件）
    ↓
即使 Context 被清理
读一下日志就知道状态在哪
```

## 第六章：为什么这套系统有效（3分钟）

### 6.1 三个关键设计

**设计一：Trigger > Memory**

知识不是"存着等用"，而是声明式的。

```
错误：MEMORY.md 里写"aigcmd.com = openclawDLX"
正确：[P-001] 部署前必须确认项目归属
     触发条件: 任何部署操作
```

**设计二：你的判断是最终标准**

```
用户说3次"有效" → verified
用户说1次"不对" → degraded

不是 AI 自己判断
不是规则自动推断
是你的判断决定原则的命运
```

**设计三：双重保险**

```
第一层：每次对话里处理完
第二层：Heartbeat 后台兜底

你不主动看 Heartbeat
但它一直在跑
出问题的时候它能恢复
```

### 6.2 传统方法 vs 这套系统

| 对比项 | 传统方法 | 这套系统 |
|--------|---------|---------|
| 纠正次数 | 3-4次才能改掉 | 1次确认就生效 |
| 记忆持久化 | Context 满就没 | 文件持久化 |
| 执行保障 | "记住了"不管用 | 触发条件自动执行 |
| 验证机制 | 无 | principle 状态跟踪 |
| 进化速度 | 缓慢 | 用户确认即进化 |

## 第七章：代码与配置（3分钟）

### 7.1 Session Startup 配置

在 `AGENTS.md` 中：

```markdown
## Session Startup

Before doing anything else:
1. Read `SOUL.md` — this is who you are
2. Read `USER.md` — this is who you're helping
3. Read `MEMORY.md` — projects, configs, active principles
4. Read `memory/日志/YYYY-MM-DD.md` — 今日工作摘要
5. Read `principles.md` — scan and match trigger conditions
   Apply matching principles BEFORE executing actions.
```

### 7.2 Heartbeat 配置

在 OpenClaw config 中：

```json
{
  "agents": {
    "defaults": {
      "heartbeat": {
        "every": "4h",
        "target": "last"
      }
    }
  }
}
```

`HEARTBEAT.md` 内容：

```markdown
## 心跳检查（每4小时一次）

### 检查项目
1. 每日日志 — 检查是否存在，无则创建
2. Principles 状态 — verifying → verified
3. Incidents — 未处理项
4. 异常 — 需要立即处理的问题

### 报告格式
```
[心跳] 系统正常 | principles: X verified | 待处理: Y
```

如果一切正常，发送简短报告。
```

### 7.3 完整文件结构

```
~/.openclaw/workspace/
├── MEMORY.md              ← 项目状态、配置、用户偏好
├── principles.md           ← 触发式行为准则
├── incidents.md            ← 事件记录
├── HEARTBEAT.md           ← 心跳检查逻辑
├── AGENTS.md              ← Session 启动配置
├── SOUL.md                ← AI 身份
├── USER.md                ← 用户信息
└── memory/日志/
    └── YYYY-MM-DD.md      ← 每日摘要
```

## 第八章：常见问题（2分钟）

**Q1：这套系统适合谁？**

适合任何使用 AI 助手并希望它"越来越懂你"的人。尤其是：
- 重复性工作多
- 对 AI 有特定要求
- 希望 AI 能记住你的习惯和偏好

**Q2：需要多少文件？**

最少 3 个核心文件：
- `MEMORY.md`
- `principles.md`
- `incidents.md`

其他都是可选的增强。

**Q3：如果 principle 错了怎么办？**

用户说"不对" → 降级为 incident → 留在文件末尾作为参考

**Q4：Heartbeat 会浪费 token 吗？**

每4小时一次，非常轻量。只有在发现问题时才会 alert你。
